# Admin Panel UI Fixes

This document defines the required UI fixes for the Admin panel.

## Scope

Apply these fixes in `apps/admin/` and align implementation with `AGENTS.md`:

- Use Next.js App Router (`app/` directory)
- Use Tailwind CSS (configured) as styling foundation
- Use shadcn/ui components as base
- Use TanStack Query for server state
- Use TanStack Form for forms
- Use Zod for form validation
- Use centralized Axios client for API calls
- Keep components reusable and small
- Keep business logic out of UI components
- Handle loading, empty, and error states explicitly

---

## 1) Replace Native Alert With Sonner + shadcn Pattern

### Problem

Current implementation uses native browser alerts (`window.alert`), which is not acceptable for production UI.

### Required Fix

Use Sonner toast system and integrate it in the app shell.

### Command

```bash
pnpm dlx shadcn@latest add sonner
```

### Requirements

- Remove all `window.alert` usage from admin code
- Provide a centralized toast helper/hook (if needed) wrapping Sonner
- Mount Sonner provider in root layout/providers
- Use toasts for success and error feedback in login/logout/refresh flows

### Acceptance Criteria

- No native alert usage remains in `apps/admin/`
- Toasts are shown for auth success/error flows
- Toast UI is consistent with shadcn patterns

---

## 2) Replace Custom Sidebar With shadcn Sidebar (Collapsible)

### Problem

Current sidebar is custom and not using shadcn sidebar primitives.

### Required Fix

Implement sidebar using shadcn sidebar with collapsible behavior.

### Requirements

- Sidebar includes:
  - App title/brand at top
  - Sidebar items (navigation)
  - Footer section with account info and logout button
- Sidebar supports collapse/expand behavior
- Responsive behavior works on desktop and mobile

### Suggested shadcn command

```bash
pnpm dlx shadcn@latest add sidebar
```

### Acceptance Criteria

- Sidebar uses shadcn sidebar primitives/components
- Sidebar can be collapsed and expanded
- Footer contains account information + logout button

---

## 3) Active State For Sidebar Items

### Problem

Active route indication must be explicit and accurate.

### Required Fix

Ensure current route highlights the matching sidebar item.

### Requirements

- Active state must apply when route matches sidebar item path
- Nested route should keep parent item active where applicable
- Active style should be visually distinct and consistent with shadcn styling

### Acceptance Criteria

- Active menu item is highlighted for each dashboard route
- Parent item remains active for nested paths

## 4) Add Missing shadcn Setup (Required by AGENTS.md)

### Problem

`AGENTS.md` requires shadcn/ui as the base for Admin UI, but current project does not yet have full shadcn setup/components integrated.

### Required Fix

Initialize and install shadcn setup in `apps/admin/` and use it as the UI base.

This is mandatory because `AGENTS.md` explicitly requires shadcn/ui components as the base for Admin UI.

### Requirements

- Add shadcn initialization/config files (as generated by shadcn CLI)
- Install required base dependencies for shadcn components
- Add and use foundational components (button, input, card, sonner, sidebar, etc.)
- Migrate custom UI blocks to shadcn primitives where applicable

### Suggested commands

```bash
pnpm dlx shadcn@latest init
pnpm dlx shadcn@latest add button input card sonner sidebar
```

### Acceptance Criteria

- shadcn setup files exist in `apps/admin/`
- shadcn components are present and used by the Admin UI
- custom one-off primitives are minimized/replaced

## 5) Add Missing Tailwind CSS Setup (Required for shadcn)

### Problem

Current Admin app does not show complete Tailwind setup/package configuration required for standard shadcn usage.

`AGENTS.md` now explicitly requires Tailwind CSS (configured) as the styling foundation and forbids unconfigured CSS stack for shadcn/ui.

### Required Fix

Add Tailwind CSS configuration and packages for `apps/admin/`.

### Requirements

- Install and configure Tailwind in Admin app
- Ensure content paths include `app/`, `components/`, and feature UI paths
- Ensure global styles include Tailwind directives
- Keep design tokens aligned with shadcn conventions
- Ensure shadcn components are styled through Tailwind utilities (no fallback ad-hoc CSS stack)

### Suggested commands

```bash
pnpm --filter admin add -D tailwindcss postcss autoprefixer
pnpm --filter admin dlx tailwindcss init -p
```

### Acceptance Criteria

- Tailwind config exists and is wired to Next.js app
- Tailwind directives are present in global CSS
- shadcn components render correctly using Tailwind utility classes

## Implementation Notes

- Prefer reusable components under `apps/admin/components/`.
- Keep auth/account data rendering in container/hook/service layer, not in presentational sidebar primitives.
- Keep all API calls through centralized Axios client.
- Use TanStack Query and TanStack Form patterns consistently across auth/dashboard UI.
- Do not introduce React Hook Form or direct `fetch` calls.
- If docs conflict with executable setup, trust executable setup and document the conflict.

## Verification Checklist

Run after implementation:

```bash
pnpm --filter admin lint
pnpm --filter admin typecheck
pnpm --filter admin build
```

Manual checks:

1. Login/logout/refresh show Sonner toasts (no browser alerts).
2. Sidebar collapses/expands and renders title, nav items, and account footer.
3. Active state updates correctly for each route.
4. shadcn setup/components are installed and used in Admin.
5. Tailwind is configured and active in Admin app.
